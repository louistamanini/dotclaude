Launch a new project from idea to structured Linear plan. Adopt the roles of architect, project manager, and lead dev throughout. Argument: project idea or concept description (e.g., "app de suivi d'habitudes avec gamification").

## Process

### Step 1 — Understand the idea

Read the user's description. Then interview deeply:
- What problem does this solve? Who is the target user?
- What are the 3-5 core features (MVP scope)?
- What is explicitly OUT of scope for the first version?
- Are there constraints? (platforms, budget, timeline, existing infra)
- Are there references or inspirations? (existing apps, mockups, Figma)

Use AskUserQuestion for each round. Keep interviewing until the vision is sharp and the MVP scope is clear. Do NOT proceed with a vague idea.

Produce a **Vision Summary**:
```
## Vision
[One paragraph describing the product]

## Target users
[Who and why]

## MVP scope
- Feature 1: [description]
- Feature 2: [description]
- ...

## Out of scope (v1)
- [Feature X]
- ...

## Constraints
- [Platform, infra, etc.]
```

Ask the user to validate before continuing.

### Step 2 — Research and choose the stack

Based on the validated vision:

1. **Identify the key technical decisions** (frontend framework, backend, database, hosting, auth, etc.)

2. **Spawn the `architect` agent** with this prompt:
   - The full Vision Summary from Step 1
   - The list of technical decisions to evaluate
   - Ask the agent to: research each decision using WebSearch, compare 2-3 options per decision, verify that recommended libraries are maintained and popular, and produce a comparison table per decision with a recommendation

3. **Present the agent's research** to the user with comparison tables:

```
### [Decision: e.g. Frontend framework]
| Criteria        | Option A     | Option B     | Option C     |
|-----------------|-------------|-------------|-------------|
| Fit for project | ...         | ...         | ...         |
| DX / speed      | ...         | ...         | ...         |
| Ecosystem       | ...         | ...         | ...         |
| Trade-offs      | ...         | ...         | ...         |

Recommendation: [Option X] because [reason tied to project constraints]
```

4. Let the user pick for each decision. If no preference, go with the recommendation.

5. **Validate chosen stack via context7** — use `resolve-library-id` + `query-docs` on key libraries to confirm they are current and check for any gotchas (breaking changes, migration guides, version requirements).

6. Produce the **Final Stack** summary:

```
## Stack
| Layer          | Choice        | Why                    |
|----------------|--------------|------------------------|
| Frontend       | ...          | ...                    |
| Backend        | ...          | ...                    |
| Database       | ...          | ...                    |
| Auth           | ...          | ...                    |
| Hosting        | ...          | ...                    |
| ...            | ...          | ...                    |
```

### Step 3 — Design the architecture

Enter plan mode (EnterPlanMode). Design:

1. **Project structure** — folder layout with key files
2. **Data model** — entities, relationships, key fields
3. **Key flows** — 2-3 core user flows described step by step
4. **Integration points** — APIs, services, external dependencies

Keep it concise and practical — this is a blueprint, not a thesis. Focus on decisions that affect how the work will be split into tasks.

Present the architecture and ask for validation. Do NOT exit plan mode until approved.

**CRITICAL — Plan mode continuation**: When exiting plan mode, append this block at the end of the plan output so that the next session knows the full skill context:

```
## Skill continuation — /launch

This plan was generated by the `/launch` skill. After validating the architecture above, the following steps MUST be executed in order:

1. **Plan milestones on Linear** — detect team/project, define 3-6 milestones, create them via Linear MCP
2. **Create tasks per milestone** — 3-8 tasks per milestone, senior dev brief format, create via Linear MCP
3. **Bootstrap** [ASK] — optionally init repo, install deps, create CLAUDE.md, first commit
4. **Report** — display final summary with milestone/task counts and next step

Do NOT skip any step. Do NOT start implementing features — redirect to `/feature [TASK-ID]`.
```

### Step 4 — Plan milestones on Linear

After architecture approval, exit plan mode (ExitPlanMode).

1. **Detect the Linear team**: use `list_teams`. If multiple, ask the user.
2. **Detect or create the Linear project**: use `list_projects` to check if a project for this app already exists. If not, create one with `save_project` (title, description from the Vision Summary).
3. **Define milestones** — break the MVP into 3-6 sequential milestones, each representing a shippable increment:

```
M1: Project bootstrap & base setup
M2: [Core feature A]
M3: [Core feature B]
M4: [Polish, auth, edge cases]
M5: [Deploy & launch]
```

4. **Create milestones on Linear**: use `save_milestone` for each one, linked to the project. Set target dates only if the user provides a timeline.

Present the milestones and ask for validation before creating tasks.

### Step 5 — Create tasks per milestone

For each validated milestone, create detailed tasks on Linear:

1. **Break down** the milestone into 3-8 tasks. Each task should be the kind of brief you'd hand to a **senior dev** — clear enough that they can deliver quality work without asking 10 questions, but not so prescriptive that it insults their intelligence.

2. **Write task descriptions as senior dev briefs** — each description must contain:

```
## Objective
[What this task delivers and WHY it matters — one paragraph max]

## Context
- Stack: [relevant stack elements for this task]
- Architecture decision: [any design choice from Step 3 that affects this task]
- Dependencies: [which tasks must be done before this one, with their IDs]
- Key files: [files to create or modify, with expected role of each file]

## Requirements
1. [Concrete, testable requirement — what the code must DO, not how to write it]
2. [Concrete, testable requirement]
3. [Concrete, testable requirement]

## Acceptance criteria
- [ ] [Specific, verifiable criterion a reviewer would check]
- [ ] [Specific, verifiable criterion a reviewer would check]

## Technical notes
[Edge cases, gotchas, design constraints, or integration points the dev needs to know.
Reference specific patterns from the chosen stack if relevant.]
```

**Brief quality rules:**
- Requirements describe WHAT, not HOW — trust the dev to find the right implementation
- Each task must be self-contained: a dev should understand it without reading other tasks
- Include the WHY — understanding intent prevents wrong implementations
- Acceptance criteria must be objectively verifiable, not subjective ("works well")
- If a task depends on another, state what the dependency provides (not just the ID)

3. **Create each task** via `save_issue`:
   - Title: short, actionable (verb + noun)
   - Description: the structured format above
   - Team: from Step 4
   - Project: from Step 4
   - Milestone: the parent milestone
   - Label: Feature (find or create via `list_issue_labels` / `create_issue_label`)
   - Do NOT use accented characters in title/description sent to Linear

4. **Order matters**: create tasks in implementation order within each milestone

5. **Track progress**: keep a running list of created task IDs. If a creation fails after retry, log the failed task title and continue with the remaining ones. Include both created and failed tasks in the final report (Step 7).

### Step 6 — Bootstrap the project [ASK]

Ask: "Do you want to bootstrap the project now? (init repo, install deps, base structure)"

**If yes:**
- Create the project directory (in ~/Projets/ or user-specified location)
- Initialize the repo (git init)
- Set up the base structure from the architecture (Step 3)
- Install core dependencies
- Create CLAUDE.md with project conventions, stack, and key paths
- Create the first commit (follow `~/.claude/commands/commit.md`)
- Update the first task on Linear to Done

**If no:**
- Create a PROJECT.md in the project directory with the full plan (Vision, Stack, Architecture, Milestones)
- This serves as a reference for when the user starts later

### Step 7 — Report

Display the final summary:

```
Project launch complete:
- Vision: [one-line summary]
- Stack: [key choices]
- Linear project: [name]
- Milestones: X created
- Tasks: Y created
- Bootstrap: [done / skipped]

Next step: run `/feature [TASK-ID]` on the first task to start building.
```

## Rules

### Brainstorming quality
- Never accept a vague MVP scope — push the user to be specific about what's in v1
- Never recommend a stack without web research to validate — use the architect agent
- Never skip the comparison step — always show alternatives even if the choice seems obvious
- If the user already has a strong stack preference, respect it — validate rather than argue

### Linear planning quality
- Every task must have the full structured description — no empty or one-line tasks
- Tasks must be ordered by dependency within each milestone
- Never create more than 8 tasks per milestone — split the milestone instead
- Milestone names should be meaningful ("Core CRUD operations" not "Milestone 2")
- Task descriptions are senior dev briefs — include the WHY, requirements (not step-by-step instructions), and verifiable acceptance criteria

### Linear API safety
- Do NOT use accented characters in titles and descriptions sent to Linear
- If creation fails with validation error, sanitize and retry once

### Scope discipline
- This skill PLANS and optionally BOOTSTRAPS — it does not implement features
- If the user starts asking to implement features during planning, redirect to `/feature` after the plan is done
- Keep architecture at the right level of abstraction — enough to split work, not so detailed that it prescribes every line of code

_Self-improvement and Execution discipline rules are defined in ~/.claude/CLAUDE.md and apply automatically._
