Implement a feature from A to Z. Argument: feature description, Linear issue ID (e.g. FLO-42), or Linear issue URL.

## Process

### Step 1 — Source the feature

Detect the input type from the argument:

**If the argument is a Linear issue ID or URL:**
- Extract the issue ID
- Use Linear MCP `get_issue` to fetch: title, description, comments, labels, assignee, status
- Display a summary (ID, title, status, description, relevant comments)
- Ask: "Is this the right issue? Anything to clarify or add before we start?"
- If the user adds clarifications, incorporate them into the working spec

**If the argument is a text description:**
- If the description is short or ambiguous, interview the user:
  - Use AskUserQuestion to dig into technical implementation, edge cases, UI/UX implications, and tradeoffs
  - Ask about things the user might not have considered — do not ask obvious questions
  - Keep interviewing until no open questions remain
- If the description is detailed enough, verify with targeted questions:
  - Expected behavior and user-facing result?
  - Edge cases and error states?
  - Constraints (stack, performance, design system, existing patterns)?
  - Reference (mockup, existing similar feature, PR)?
- Summarize the complete spec and ask for confirmation

**If no argument:** ask the user what they want to build.

Do NOT start planning until the spec is confirmed.

### Step 2 — Options checkpoint

Ask upfront, all at once:

- **Architect analysis**: Should the `architect` agent analyze the codebase and propose 2-3 approaches with trade-offs? Recommended for complex features, new systems, or when an external library is involved.
- **Simplify pass**: Should a code-simplifier agent run before review? Recommended for large features (10+ files). Skipping saves tokens — the reviewer will still catch issues.
- **Tests**: Should tests be written alongside the implementation?
- **Linear issue** [only if NOT sourced from Linear]: Should a Linear issue be created at the end to document this feature?

Collect all answers before moving on.

### Step 3 — Explore and plan

Enter plan mode (EnterPlanMode). In plan mode:

**If the user requested architect analysis:**
- Spawn the `architect` agent with the full feature spec
- The agent will explore the codebase and propose 2-3 approaches with trade-offs, a recommendation, and a detailed implementation plan
- Present the agent's output and ask the user to pick an approach
- Use the chosen approach as the basis for the execution plan

**In all cases:**
- Identify all files to create or modify with exact paths and line numbers
- Produce a clear, step-by-step execution plan

Do not exit plan mode until the user explicitly approves the plan. If the user requests changes, revise and re-present. Only exit plan mode (ExitPlanMode) once the plan is approved.

**CRITICAL — Plan mode continuation**: When exiting plan mode, append this block at the end of the plan output so that the next session knows the full skill context:

```
## Skill continuation — /feature

This plan was generated by the `/feature` skill. After implementing the plan above, the following steps MUST be executed in order:

1. **Simplify** [if requested] — Spawn the `code-simplifier` agent on all modified files
2. **Review** — Read and follow `~/.claude/commands/review.md` on all modified files
3. **Tests** [if requested] — Write tests for core logic, then follow `~/.claude/commands/verify.md`
4. **Playwright audit** — Ask: "Do you want a Playwright E2E audit?" If yes, follow `~/.claude/commands/audit-feature.md`
5. **Commit** — Read and follow `~/.claude/commands/commit.md`
6. **Linear update** [if sourced from Linear] — Use `save_issue` to set status to Done + comment (what was implemented, key files, commit hash)
7. **Linear create** [if requested at options] — Follow `~/.claude/commands/create-issue.md`

Do NOT skip review — it is mandatory.
Do NOT commit while review blockers remain unresolved.
```

**If sourced from a Linear issue:** update the issue status to **In Progress**.

### Step 4 — Implement

Execute the approved plan step by step, following existing conventions (naming, structure, imports). If a step reveals unexpected complexity or a blocker, stop and inform the user before continuing.

### Step 5 — Simplify [CONDITIONAL]

If the user requested a simplify pass in Step 2:
- Spawn the `code-simplifier` agent on all files modified during this feature
- The agent will refine the code for clarity, consistency, and maintainability while preserving exact functionality
- Let it complete its full pass without interruption

### Step 6 — Review

Read and follow `~/.claude/commands/review.md`. The scope is all files modified during this feature.

### Step 7 — Verify and test [CONDITIONAL]

If the user requested tests in Step 2:
- If the feature has no test coverage, write tests for the core logic first
- Then read and follow `~/.claude/commands/verify.md` to run all available checks and fix failures

### Step 8 — Playwright audit [ask here]

After the code is solid, ask:

> "Do you want a Playwright E2E audit on this feature?"

If yes, read and follow `~/.claude/commands/audit-feature.md` with the feature flow as the test description.

### Step 9 — Commit

Read and follow `~/.claude/commands/commit.md`.

If sourced from a Linear issue, include the issue ID in the commit message if the project's commit style supports it (e.g. `feat: add X [FLO-42]` or `feat(FLO-42): add X`). If the style has no issue reference pattern, omit it.

### Step 10 — Linear [CONDITIONAL]

**If sourced from a Linear issue:**
- Use Linear MCP `save_issue` to update the issue status to **Done** (or the equivalent closing status in the team's workflow)
- Add a comment on the issue summarizing: what was implemented, key files modified (with paths), commit hash

**If the user requested a new Linear issue in Step 2:**
- Read and follow `~/.claude/commands/create-issue.md` using the feature description and implementation context as input

## Rules

- Never start implementing before the plan is approved (Step 3)
- Never commit while review blockers remain unresolved (Step 6)
- Never skip the review step — it is not optional
- If tests are requested and fail, fix before committing — never commit with failing tests
- At each conditional step, honour exactly what the user answered in Step 2 — do not re-ask
- If sourced from Linear, always update the issue at the end (Step 10) — this is not optional
- If the Linear issue cannot be fetched (wrong ID, permission error), stop and ask the user

_Self-improvement and Execution discipline rules are defined in ~/.claude/CLAUDE.md and apply automatically._
